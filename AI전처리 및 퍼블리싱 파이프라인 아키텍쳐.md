# 가이드 AI 전처리 및 퍼블리싱 파이프라인 아키텍처

## 1. 핵심 설계 철학 및 아키텍처 방향
* **역할 분리 (Separation of Concerns):** 문서를 '순수 정보성 가이드(Fact)'와 '핀포인트 공략(Context)'으로 이원화합니다. 팩트는 인간이 SDE를 기반으로 구축하고, AI와 위키 파싱은 철저히 맥락 중심의 공략 문서에만 집중합니다.
* **통제된 AI 개입 (Deterministic Control):** AI의 비결정론적 한계를 통제하기 위해, 문서 분할은 기계적(검색엔진/룰 기반)으로 수행하고 AI는 정해진 규격(JSON 태그 풀, 통문서 분석) 내에서만 작동하도록 강제합니다.
* **비동기 분산 처리 (SQS Driven):** 커뮤니티 백엔드와 Discord 봇 간의 모든 상태 동기화 및 알림은 AWS SQS를 거쳐 데이터 유실과 데드락을 원천 차단합니다.

---

## 2. 데이터 인제스천 및 퍼블리싱 파이프라인 6단계

### Step 1: 크롤링 및 자산 링크 유지 (Ingestion)
E-UNI 위키(CC BY-SA)에서 텍스트 위주로 기계적 파싱을 수행하여 DB에 '핀포인트 공략' 카테고리로 저장합니다. 이미지는 스토리지 부담과 파일 업로드 파이프라인의 복잡도를 줄이기 위해 핫링킹(원본 링크) 상태를 유지하며, 추후 인간 검토자가 한글 클라이언트 캡처본으로 교체하도록 유도합니다.

### Step 2: AI 1차 번역 및 다이렉트 임시저장 (Drafting)
파싱된 데이터를 경량 모델(Gemini 2.5 Flash Lite)과 SDE 번역 스킬(Function Calling)을 활용해 1차 번역합니다. 에디터(BlockNote) 단의 파싱 에러를 막기 위해 마크다운을 배제하거나 호환 규격만 사용하도록 강제합니다. 완료된 초안은 라이브 DB가 아닌 임시 대기열(Draft)에 다이렉트로 저장되며, SQS를 통해 Discord로 편집 요청(Webhook/API)을 발송합니다.

### Step 3: 휴먼 인 더 루프 (Human-in-the-Loop)
유저가 커뮤니티에서 임시 문서를 직접 수정합니다. 무거운 Git 대신 `diff-match-patch`나 `jsdiff`를 도입하여 가볍고 직관적인 텍스트 Diff(빨간색/초록색 하이라이팅)를 제공합니다. 시스템은 3인 이상의 독립적인 검토/승인이 완료되어야만 문서를 확정하며, 도중 수정이 발생하면 승인 카운트는 즉시 초기화됩니다. 변경 상태는 Discord 스레드와 실시간 동기화됩니다.

### Step 4: 포매팅 (Formatting)
3인 교차 검증이 끝나면, Vercel AI 호환성을 고려해 GPT 기반의 BlockNote AI를 호출합니다. 이 단계의 AI는 내용을 절대 수정하지 않고 문서의 서식과 에디터 구조만 깔끔하게 정리하여 최종 저장합니다.

### Step 5: 태깅 및 동적 링킹 (Tagging & Dynamic Linking)
저장된 문서를 기계적으로 분리한 뒤, Gemini를 호출하여 전체 문맥을 읽고 태그를 추출합니다. 이때 환각을 막기 위해 API `temperature=0`으로 고정하고, "사전 제공된 JSON 태그 배열 내에서만 선택할 것"을 시스템 프롬프트로 강제합니다. 추출된 태그를 기반으로 프론트엔드에서는 공략 글(Context) 열람 시 연관된 스펙 글(Fact)을 정적 링크/툴팁 형태로 자동 노출합니다.

### Step 6: 패치노트 기반 자동화 추적 (Automated Patch Tracking)
백엔드가 최신 패치노트를 구독하면 Gemini가 변경된 객체를 태깅합니다. 백엔드는 해당 태그를 포함하는 기존 문서를 필터링하고, 문서 ID와 제목 메타데이터가 강제로 심어진 '텍스트 청크'를 Gemini에게 전송합니다(SSO 우회 목적). AI가 논리적 근거와 함께 수정 필요 판정을 내리면, 이 결과를 SQS에 태워 Discord에 수정 요망 이슈(스레드)를 자동 생성합니다.
